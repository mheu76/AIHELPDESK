# 06. 하이브리드 RAG 아키텍처 및 구현 스펙

## 1. Situation (현황)
* **기존 구조:** Claude Embeddings API + ChromaDB 기반 순수 의미론적 검색(Dense Retrieval).
* **문제점:** 사내 고유 명사, 특정 에러 코드(예: ERP 에러 404), 사번 등 정확한 키워드 매칭이 필수적인 검색에서 문서 누락 및 정확도 급감 발생.

## 2. Complication (핵심 판단 및 반론)
* **판단:** 별도의 무거운 검색 엔진을 즉시 도입하기보다, 기존 인프라인 PostgreSQL의 전문 검색(Full-Text Search)을 활용하여 '하이브리드 RAG'를 구축하는 것이 서버 리소스와 운영 복잡도를 최소화하는 최적의 방안임.

## 3. Resolution (단계별 구현 방안)

### 1단계: PostgreSQL Full-Text Search 세팅
* `kb_documents` 테이블에 키워드 검색용 `tsvector` 컬럼 추가.
* 한국어 형태소 분석을 보완하기 위해 `pg_trgm` (N-gram) 또는 `pgroonga` 확장 모듈 설치.
* 텍스트 검색 속도 향상을 위한 GIN 인덱스 생성.

### 2단계: 문서 업로드 파이프라인 이원화
* 관리자 문서 업로드 시, 분할된 텍스트 청크를 두 데이터베이스에 병행 저장.
    * **의미 저장 (Semantic):** 임베딩 모델 거친 후 ChromaDB 적재.
    * **키워드 저장 (Keyword):** 원본 텍스트를 PostgreSQL 청크 테이블에 업데이트.

### 3단계: FastAPI 검색 로직 병렬 처리
* 사용자 질의 인입 시 `asyncio.gather`를 활용해 두 DB를 비동기로 동시 호출.
    * **ChromaDB 질의:** 벡터 유사도 기반 Top-10 문서 추출.
    * **PostgreSQL 질의:** 키워드 매칭 기반 Top-10 문서 추출.

### 4단계: Reciprocal Rank Fusion (RRF) 병합 알고리즘 적용
* 점수 체계가 다른 두 검색 결과를 공평하게 병합하기 위해 순위(Rank) 기반 재정렬 수행.
* 최종 RRF 점수 기준 상위 3개(Top-3) 문서를 선정하여 LLM 프롬프트에 주입.

**[RRF 점수 산출 수식]**
$$RRF\_Score = \frac{1}{k + rank_{vector}} + \frac{1}{k + rank_{keyword}}$$
*(k: 극단적인 순위 편향을 막기 위한 보정 상수, 일반적으로 60 적용)*
